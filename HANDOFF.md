## 1. 現状サマリー
- **動く MVP／初期導入版**。本番稼働中（Vercel / `main`）、デモアカウントで全機能操作可。
- E2E 回帰（`qa/`）で 20 圃場シナリオを含めクラッシュ 0 を確認済み。
- ただし「フルスタックが即・無改修で運用継続できるプロダクト」ではない。下記の負債を理解した上で引き継ぐこと。

## 2. まず 30 分でやること（キャッチアップ手順）
1. [README.md](./README.md) → [ARCHITECTURE.md](./ARCHITECTURE.md) を読む。
2. `npx serve .` でローカル起動 → デモアカウントでログイン → 全ページを触る。
3. `cd qa && npm install && npm run test:farm` で E2E を1回通す（緑=`errorCount:0`）。
4. DevTools の Application → Local Storage を開き、`farm_*_<farmId>` キーの中身を眺める（データの実体はここ）。

## 3. 技術的負債（優先度順）

### 🔴 P0: 農場データが localStorage（本番化の最重要論点）
- 記録・ロット・収穫など**業務データがブラウザ内**。多端末同期不可・多人数同時編集で不整合・消失リスク・サーババックアップ無し。
- **推奨**: Supabase に業務テーブルを新設し、`app.js` の `useFPS` を Supabase 読み書き（+ローカルキャッシュ）へ置換。
  - `useFPS(key, initial)` が唯一の永続化境界なので、ここを差し替えれば全画面に波及できる設計にはなっている（不幸中の幸い）。
  - RLS を org/farm 単位で設計。既存の `farm_members.farm_ids` を流用。
  - 写真は Supabase Storage へ（現状 base64/localStorage・4MB 上限）。

### 🟠 P1: モノリシックな無ビルド・フロントエンド
- `js/components.js` が**約 15,000 行・870KB の 1 ファイル**、JSX 無し、モジュール無し。
- **これは「一人で爆速に回す」ための最適化でもある**。複数人開発 or 引き継ぎが始まる時に着手する前提。
- **具体手順**: → [MIGRATION.md](./MIGRATION.md)（Vite 導入〜段階的分割〜JSX 化を Strangler で安全に）。
- **迷子対策**: → [COMPONENT_MAP.md](./COMPONENT_MAP.md)（106 関数の「機能→行範囲」索引。分割の切り出し単位にそのまま使える）。

### 🟡 P2: 設定・依存の管理
- `config.js` に Supabase URL / anon key 直書き → ビルド導入時に `.env`（`VITE_*`）へ。
- CDN の一部が `@latest`（Tabler Icons・Leaflet）→ バージョン固定でないと突然壊れうる。ビルド化時に npm 依存へ寄せる。
- ルートに `package.json` が無い（qa 配下のみ）。ビルド導入とセットで用意。

### 🟢 P3: コード内の既知の不整合・デッドコード
- `FieldPerformancePage` の資材コストタブは農薬使用量に **`disposal_amount`（廃棄量）** を使っており、本来の原液使用量とズレる。
  「圃場まとめ」(`FieldSummaryPage`) は正しいモデル（散布液量÷希釈倍率）。**片方に統一すべき**。
- `PlanCompareCard`（開発モデル比較）は UI から外したが**関数定義は残置**（dead code、削除可）。
- `LotsStep2DebugPanel` は定義のみで未レンダリング（削除可）。
- 作物名 → カテゴリの対応が緩い。例: `crop='米'` はカテゴリ名 `水稲` と一致せず既定 `葉物野菜` に落ちる（`getCropCategory`）。収穫予測のしきい値選択に影響。マッピング表 or 圃場作成時のカテゴリ明示で改善を。

### テスト
- 単体テストは無い。`qa/` の実ブラウザ E2E のみ（CI 未設定）。ビルド化後に Vitest 等 + GitHub Actions を推奨。

## 4. 未確定の業務仕様（今福さん／中川さんに要確認）
コード内コメントにも `要確認` として点在。実データ投入前に確定が必要:
- 肥料は **袋管理か kg 管理**か（現状 kg 統一・`weight_per_bag_kg` で袋⇔kg 変換可）。
- 重さ不明の肥料品目（エコレット〜BOSO ペレット等）は暫定 **20kg 仮置き**。
- 価格未入力の肥料品目の扱い（原価計算に影響）。
- **圃場番号の表記ゆれ正規化ルール**（`FIELD_NO_NORMALIZE` / `data.js`）。3S/3N 等の統一規則。
- 米 → レタス転換（8月末〜）時の作物/カテゴリの扱い（シーズン境界は月>=9 で新シーズン）。

> 方針: 分からない入力項目は憶測で埋めず、**データ管理担当（今福さん）が実際に使って要望が出た時点で調整**する運用。

## 5. ハマりどころ（Gotchas）
- スクリプトはグローバル定義の積み上げ。`index.html` の読み込み順を崩さないこと。
- localStorage キーは必ず `<key>_<farmId>` サフィックス付き。農場を切り替えると別名前空間になる。
- 記録とロットは外部キーで繋がっていない。突合は `field_id` + 畝範囲の重なり（`parseRowRange`）。
- 保存系の演出 `celebrateSave()` は DOM 直生成。二重発火を避けるため、定植→ロット自動生成では呼ばない（日報側で発火済み）。
- 認証は実 Supabase。E2E もデモアカウントで実ログインする（オフライン不可）。
